# Анализ технических возможностей и жанра игры

## Техническая архитектура

### База данных
- **Firebird SQL** - основная СУБД для хранения игровых данных
- **Qdrant** - векторное хранилище для продвинутых поисковых возможностей
- **C# backend** - серверная логика на .NET
- **Godot 4** - игровой движок для клиентской части

### Инструменты нарратива
1. **Диалоговые окна** - встроенные в Godot UI элементы
2. **Письма** - email-клиент с системой заданий
3. **Уведомления** - системные сообщения и оповещения
4. **Системные сообщения** - логи и отладочная информация
5. **Записи в базе данных** - игровые данные как часть сюжета
6. **Терминал/консоль** - ввод SQL-запросов
7. **Справочная система** - встроенная документация
8. **Судоку** - мини-игра для разнообразия
9. **Браузер** - доступ к сайту с новостями, wiki о мире, доски с объявлениями и юмором
10. **Прогресс-система** - сохранение и загрузка состояния

### Жанр и механики
- **Обучающая игра** - обучение SQL через игру
- **Симулятор работы** - офисная рутина с элементами драмы
- **Пазл-игра** - решение задач через SQL-запросы
- **Нарративная игра** - сюжет развивается через электронную почту
- **Менеджмент** - управление временем и ресурсами

- Практические примеры
- Обратная связь

## Вопросы для синхронизации видения

### Основные вопросы
1. **Целевая аудитория**: Кто основной игрок - новички в SQL который в процессе игры обогащает свои знания о SQL в общем и особенностям Firebird в часности
2. **Тон и атмосфера**: Серьезная корпоративная симуляция с элементами техномистики
3. **Сложность**: Уровень запросов - за 90 игровых дней игрок должен пройти от самых простых команд до самых сложных - по нарастающей
4. **Сюжет**: Сюжет в основном линейный, с редкими случайными событиями немного влияющие на концовки
5. **Концовки**: Я планировал 5 концовок - в зависимости от действий игрока в ключевые моменты

### Технические вопросы
1. **Баланс**: Так как игра активно использует SQL запросы? Обучение на первом месте, но идея игры - увлекательно обучить игрока (или развлеч опытного)
2. **Темп**: Я ориентировался примерно на 10 минут реального времени, но время в игре не постоянно - исполнение запроса занимает какие-то количество времени, и чем сложнее запрос - тем больше времени он отнимет
3. **Прогрессия**: Нужно за время игры научить около 70 команд и функций (уровень upper junior)
4. **Обратная связь**: Учитывая что в терминал планируется ввести систему проверки ответа (но по игровым условностям она может давать ложно положительный результат в одном из треъ неправильных ответах) - я думал о трёх уровневой системе предупреждений: одна ошибка - просто замечание, две ошибки - штраф по времени, три ошибки - закрывается одна из концовок.
5. **Повторяемость**: На данный момент я думал о минимальной вариантивности, но открыт для предложений

### Нарративные вопросы
1. **Персонажи**: На данный момент запланировано три игровых персонажа(за которых игрок будет последовательно играть), около пока 10 безликих отправителя писем с сюжетными заданиями. Так как всё взаимодействие будет идти с Базой/Базами Данных, фактически - она главный персонаж с которым взаимодействует игрок с помощью SQL-запросов
2. **Конфликт**: Разгадать загадку НИИ и его проекта
3. **Тайминг**: Каждый игровой день приходит несколько писем с заданиями, которые надо правильно завершить до конца игрового дня. Допускаю срабатывание случайных событий, для разнообразия игрового процесса (например нужно решить определенную задачу за 2 минуты реального времени)
4. **Моральные выборы**: Должны ли быть этические дилеммы? Да, в конце каждой игровой части - на 30, 60 и 90 день будет главный моральный выбор за текущего персонажа, нужны так же мелкие развилки по ходу, но с мягким влянием
5. **Юмор**: Так как игра образрвательная, но лёгкий юмор профессионального уклона уместен

### Дизайн-вопросы
1. **Визуальный стиль**: Ретро-киберпанк
2. **Музыка и звук**: Эмбиент со звуками ламп, гудения старых компьютеров, шелчки по клавишам, у одного из игровых персонажей живёт кот
3. **Интерфейс**: Интерфейс - максимально упрощенная ОС, с монохромным экраном, и интуитивно понятными иконками, а так же подписями на кнопках 
4. **Мини-игры**: На данный момент запланирована и реализована одна - судоку, и только как внеигровое развлечение. Но открыт для предложений если миниигра будет органично интергрирована
5. **Доступность**: Если я правильно понял - вопрос про доступность для людей с ограниченными возможностями? Можно предусмотреть изменения главного цвета с зеленого на другой (для людей с нарушенным цветовосприятием), шрифт планируется и так крупный из-за специфики игрового мира, всё остальное пока не обсуждалось, но открыт для предложений.
