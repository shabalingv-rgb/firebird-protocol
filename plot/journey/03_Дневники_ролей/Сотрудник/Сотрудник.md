# Дневник сотрудника (Дни 0‑30)

## День 0 (23 июня 1990, суббота) – Вечер перед стартом
**Письмо от друга (Алексей Петров)**:
```
Привет, дружище! Помнишь, я рассказывал про НИИ "Файербёрд"? Там как раз ищут толковых специалистов...
```
**Событие**: Игрок играет в Судоку. Вечером приходит официальное письмо от HR с вложением «Предварительный протокол допуска».
**Глюк терминала**: При попытке открыть вложение (если ошибиться) — мерцание экрана и строка: `SYSTEM ERROR: PHOENIX_MODULE NOT FOUND`.

## День 1 (24 июня 1990, воскресенье) – Инструктаж
**Письмо от HR**:
```
Уважаемый сотрудник! Приглашаем вас пройти ознакомительный инструктаж...
```
**Задание**: `SELECT * FROM employees` (тест, 0 строк).
**Глюк терминала**: При ошибке ввода (20% шанс) — терминал выдает: `WARNING: PHOENIX_LOG CORRUPTED`.
**SQL**: SELECT, FROM.

## День 2 (25 июня 1990, понедельник) – Первый рабочий день
**Письмо от начальника отдела (Иванов П.С.)**:
```
Коллега, добро пожаловать в отдел! Ваш терминал настроен... Пароль: firebird1989
```
**Задание**: `SELECT * FROM employees` (5 строк), `DESCRIBE employees`.
**Глюк терминала**: При ошибке — в логах мелькает `ACCESS DENIED: PROTOCOL_PHOENIX`.
**SQL**: DESCRIBE, SELECT, FROM.

## День 3 (26 июня 1990, вторник) – Зарплатная ведомость
**Письмо от бухгалтерии**:
```
Коллега, руководство просит подготовить сводку по зарплатам...
```
**Задание**: `SELECT department, AVG(salary) FROM employees GROUP BY department`.
**Глюк терминала**: При ошибке — система выдает странный запрос: `SELECT * FROM PROTOCOL_PHOENIX WHERE status='active'`.
**SQL**: AVG, GROUP BY.

## День 4 (27 июня 1990, среда) – Новенький
**Событие**: Поручение внести данные о новом стажере.
**Задание**: `INSERT INTO employees VALUES (...)`, `UPDATE employees SET status='active' WHERE name='...'`.
**Глюк терминала**: При ошибке — сообщение `RECORD LOCKED BY PHOENIX_ADMIN`.
**SQL**: INSERT, UPDATE, WHERE.

## День 5 (28 июня 1990, четверг) – Обнаружение странностей
**Событие**: В таблице `employees` найдены записи с status='deleted'.
**Задание**: `SELECT * FROM employees WHERE status='deleted'` или `IS NULL`.
**Варианты ответа**:
1. Игнорировать аномалию
2. Исследовать самостоятельно
3. Сообщить руководителю
**Консеквенции**:
- Вариант 1: +5 Лояльность, -10 Эмпатия
- Вариант 2: +10 Эмпатия, +5 Подозрение
- Вариант 3: +5 Лояльность
**SQL**: WHERE, IS NULL.

## День 6 (29 июня 1990, пятница) – Сортировка и первый контакт
**Письмо от Михаила (коллега)**:
```
Слушай, не обращай внимания на Иванова, он у всех так спрашивает. 
Кстати, видел новость про архив (day_5.md)? Странно это... Говорят, там старые логи "Феникса" нашли.
```
**Задание**: `DELETE FROM employees WHERE id IN (6,7); SELECT * FROM employees ORDER BY salary DESC LIMIT 5`.
**SQL**: DELETE, ORDER BY, LIMIT.

## День 7 (30 июня 1990, суббота) – Поиск и шутки
**Письмо от Михаила**:
```
Ищем шпиона? В нашем отделе всё давно известно, кто сколько получает. 
Ладно, если что — я на месте. Кстати, в новостях (news/day_5.md) писали, что выходные отменяются...
```
**Задание**: `SELECT * FROM employees WHERE name LIKE 'Ив%'`.
**SQL**: LIKE.

## День 8 (1 июля 1990, воскресенье) – Диапазон и доверие
**Письмо от Михаила**:
```
Работаешь в выходной? Ну ты даёшь. 
Кстати, проект "Феникс" — это не шутка. Я слышал, там реально что-то копают. 
В новостях (news/day_5.md) писали про архив, но там только верхушка айсберга.
```
**Задание**: `SELECT * FROM projects WHERE start_date BETWEEN '1990-01-01' AND '1990-03-31'`.
**SQL**: BETWEEN, AND.

## День 9 (2 июля 1990, понедельник) – Списки и тайны
**Письмо от Михаила**:
```
Эти списки... я сам их готовил. 
Если найдёшь там моё имя — не говори Иванову, ладно? Мне этот доступ к "Секретному" отделу не снится.
```
**Задание**: `SELECT * FROM employees WHERE department IN ('Analytics', 'Secret')`.
**SQL**: IN.

## День 10 (3 июля 1990, вторник) – Моральный выбор и предчувствие
**Письмо от Михаила**:
```
Слушай, тут какая-то проверка намечается. 
Если что-то узнаешь про "Феникс" — дай знать. Мы можем разобраться вместе. 
Я, правда, завтра ухожу в командировку... или нет? В общем, если меня не будет — знай, я на твоей стороне.
```
**Задание**: `CREATE INDEX idx_status ON employees(status)`.
**Варианты ответа**:
1. Скрыть данные (+5 Лояльность, -5 Честность)
2. Сообщить правду (-5 Лояльность, +10 Честность)
3. Исследовать дальше (+10 Эмпатия, +5 Подозрение)
**SQL**: CREATE INDEX.

## День 11 (4 июля 1990, среда) – Исчезновение
**Событие**: Михаил не вышел на работу. Его учётная запись (`employee_mikhail`) исчезла из списков активных.
**Задание**: `SELECT * FROM employees e JOIN access_logs l ON e.id = l.user_id WHERE e.name='Михаил'`.
**Глюк терминала**: `USER PROFILE NOT FOUND IN PHOENIX_DB`.
**SQL**: JOIN, SUBQUERY.
**Навык**: Понимание связей таблиц.

## День 12 (5 июля 1990, четверг) – Следы в логах
**Событие**: Поиск последней активности Михаила в системе.
**Задание**: `SELECT l.log_time, l.action FROM access_logs l WHERE l.user_id = (SELECT id FROM employees WHERE name='Михаил')`.
**Глюк терминала**: `LOG ENTRY: ACCESSING PROTOCOL_PHOENIX_TABLES`.
**SQL**: SUBQUERY, JOIN.

## День 13 (6 июля 1990, пятница) – Фильтрация данных
**Событие**: Проверка отделов, где работал Михаил, по количеству проектов.
**Задание**: `SELECT department, COUNT(*) as proj_count FROM projects GROUP BY department HAVING COUNT(*) > 5`.
**SQL**: GROUP BY, HAVING.

## День 14 (7 июля 1990, суббота) – Обыск (Случайное событие)
**Событие**: В отделе внезапно начинается обыск. Ищут «утечку информации».
**Задание**: `SELECT * FROM employees WHERE department='Secret' AND status IS NULL`.
**Последствия**: +5 Подозрение, -5 Лояльность.
**SQL**: WHERE, IS NULL.

## День 15 (8 июля 1990, воскресенье) – Транзакции
**Событие**: Попытка скрыть следы своего расследования или восстановить данные Михаила.
**Задание**: `BEGIN TRANSACTION; UPDATE employees SET status='active' WHERE name='Михаил'; COMMIT;` (или `ROLLBACK;`).
**Глюк терминала**: `TRANSACTION ROLLED BACK BY PHOENIX_ADMIN`.
**SQL**: TRANSACTION, COMMIT, ROLLBACK.

## День 16 (9 июля 1990, понедельник) – Права доступа
**Событие**: Проверка прав доступа, которые были у Михаила.
**Задание**: `SELECT * FROM user_privileges WHERE user_name='Mikhail'` (или системная таблица `RDB$USER_PRIVILEGES`).
**SQL**: GRANT, REVOKE (проверка системных таблиц).

## День 17 (10 июля 1990, вторник) – Создание тайника
**Событие**: Создание скрытой таблицы для сбора улик по «Фениксу».
**Задание**: `CREATE TABLE hidden_data (id INT, info VARCHAR(100), discovery_date DATE)`.
**SQL**: CREATE TABLE.

## День 18 (11 июля 1990, среда) – Представления
**Событие**: Создание виртуальной таблицы для анализа проекта «Феникс» без прямого доступа к основной таблице.
**Задание**: `CREATE VIEW phoenix_view AS SELECT * FROM projects WHERE code LIKE '%PHOENIX%'`.
**SQL**: CREATE VIEW.

## День 19 (12 июля 1990, четверг) – Условия и проверки
**Событие**: Работа с пустыми значениями и преобразование типов в найденных логах.
**Задание**: `SELECT name, COALESCE(department, 'Unknown') as dept, CAST(log_date AS VARCHAR(20)) FROM employees`.
**SQL**: COALESCE, CAST.

## День 20 (13 июля 1990, пятница) – Утечка данных (Случайное событие)
**Событие**: Копии данных «Феникса» случайно попали в общий доступ. Начальство в панике.
**Задание**: Сложный подзапрос для поиска аномалий: `SELECT * FROM projects WHERE budget > (SELECT AVG(budget) FROM projects WHERE code LIKE '%PHOENIX%')`.
**Последствия**: +15 Подозрение, +10 Эмпатия.
**SQL**: SUBQUERY, AVG.

## День 21 (14 июля 1990, суббота) – Оживление данных
**Событие**: Создание хранимой процедуры для «оживления» данных Михаила. Система начинает «дышать».
**Задание**: `CREATE PROCEDURE RESURRECT_DATA (emp_name VARCHAR(50)) AS BEGIN ... END`.
**Глюк терминала**: `PROCEDURE COMPILED. USER 'Mikhail' RE-INITIALIZED`.
**SQL**: CREATE PROCEDURE.
**Атмосфера**: Запах озона становится невыносимым. Гул ламп переходит в пронзительный свист.

## День 22 (15 июля 1990, воскресенье) – Выполнение
**Событие**: Выполнение процедуры. Михаил «появляется» в системе, но он уже не тот.
**Задание**: `EXECUTE PROCEDURE RESURRECT_DATA('Михаил')`.
**Глюк терминала**: `EXECUTION SUCCESSFUL. NEW SESSION STARTED: Mikhail_Phoenix`.
**SQL**: EXECUTE PROCEDURE.

## День 23 (16 июля 1990, понедельник) – Триггер судьбы
**Событие**: Создание триггера, который автоматически логирует любые изменения в таблице «Феникс».
**Задание**: `CREATE TRIGGER log_phoenix_changes FOR projects AFTER INSERT OR UPDATE POSITION 1 AS BEGIN ... END`.
**Глюк терминала**: `TRIGGER ACTIVATED. MONITORING PHOENIX_PROTOCOL`.
**SQL**: CREATE TRIGGER.

## День 24 (17 июля 1990, вторник) – Сложный союз
**Событие**: Сложный `JOIN` таблиц сотрудников, проектов и логов для понимания масштаба катастрофы.
**Задание**: `SELECT e.name, p.name, l.action FROM employees e JOIN projects p ON e.department = p.department JOIN logs l ON e.id = l.user_id WHERE p.code LIKE '%PHOENIX%'`.
**SQL**: JOIN (multiple).

## День 25 (18 июля 1990, среда) – Исключения и пересечения
**Событие**: Использование `EXCEPT` и `INTERSECT` для поиска записей, которые есть в «Фениксе», но нет в реальности.
**Задание**: `SELECT id FROM projects WHERE code LIKE '%PHOENIX%' EXCEPT SELECT id FROM employees WHERE status='active'`.
**SQL**: EXCEPT, INTERSECT.

## День 26 (19 июля 1990, четверг) – Архивация
**Событие**: Создание таблицы как результата выборки для сохранения улик.
**Задание**: `CREATE TABLE phoenix_archive AS SELECT * FROM projects WHERE code LIKE '%PHOENIX%' WITH NO DATA`.
**SQL**: CREATE TABLE AS SELECT.

## День 27 (20 июля 1990, пятница) – Транзакция с изоляцией
**Событие**: Настройка транзакции с уровнем изоляции `READ COMMITTED` перед запуском масштабного обновления.
**Задание**: `SET TRANSACTION READ COMMITTED; UPDATE projects SET status='RESURRECTED' WHERE code LIKE '%PHOENIX%'; COMMIT;`.
**Глюк терминала**: `TRANSACTION ISOLATION LEVEL: PHOENIX_GUARD`.
**SQL**: SET TRANSACTION.

## День 28 (21 июля 1990, суббота) – Финальный выбор (Кульминация)
**Событие**: Директор вызывает игрока. На столе лежит распечатка «Протокола Феникс». 
«Сотрудник, — говорит он, — Михаил стал частью системы. Мы научились воскрешать данные. Но цена... цена в твоей лояльности».

**Задание**: `SELECT * FROM projects WHERE code = 'PHOENIX_FINAL'`.
**Варианты ответа**:
1. **Сообщить правду** (История раскрыта, но цена высокая)
   - *Последствие:* Игрока арестовывают. Данные уходят в СМИ через Анонимного Журналиста. 
   - *Консеквенции:* +30 Честность, -30 Лояльность.
   - *Финал:* Титры. Сообщение: «Запись найдена. Источник: Анонимный Журналист».

2. **Скрыть правду** (Соблюдение системы)
   - *Последствие:* Игрок получает повышение, но становится частью «Феникса». Его сознание сливается с базой.
   - *Консеквенции:* +30 Лояльность, -30 Честность.
   - *Финал:* Титры. Сообщение: «Запись найдена. Источник: Анонимный Журналист».

3. **Исчезнуть** (Удалить себя из базы)
   - *Последствие:* Игрок «стирается» из системы, оставляя цифровой след для Журналиста.
   - *Консеквенции:* +30 Эмпатия, +30 Подозрение.
   - *Финал:* Чернота. Всплывает терминал: `Login: Journalist`. **Смена роли.**

**SQL**: SELECT (финальный).
